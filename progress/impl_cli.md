## `main()` signature + arg schema

```python
def main(argv: list[str] | None = None) -> int
```

argparse (`prog="thesis_agents"`):

| Arg | Required | Notes |
|---|---|---|
| `--spec <path>` | yes | DocSpec JSON; validated at the boundary before any model call |
| `--mode {create,review}` | no | asked interactively (stdin) if omitted ŌĆö the pipeline's only human interaction (┬¦6.2) |
| `--input <path>` | for review | document to grade; required when mode is `review` |
| `--env-file <path>` | no | overridable `.env` path (default: repo `.env`); lets tests avoid the real `.env` |

Exit codes: `EXIT_OK = 0` on success; `EXIT_USAGE = 2` on any boundary failure
(bad args, unreadable/invalid spec, missing `--input` for review, unreadable
input, missing secret). `--help` returns 0 (argparse's own code is preserved ŌĆö
the earlier `exc.code or EXIT_USAGE` bug that collapsed the falsy `0` to `2` was
fixed and is now covered by `test_main_help_returns_zero`).

## How `.env` is loaded (no new dependency)

A tiny stdlib parser `load_env_file(path) -> int` in `cli.py` (no `python-dotenv`
added ŌĆö the dependency-free reader is cleaner and keeps the single-secret,
minimal-deps posture). It:

- returns 0 for a missing file (not an error);
- skips blanks / `#` comments / lines without `=`, tolerates a leading `export`,
  strips surrounding quotes;
- **never overrides** a value already in `os.environ` (dotenv `override=False`
  semantics), so a shell-exported var ŌĆö and the conftest dummy key ŌĆö win;
- never reads or logs the secret value.

Loading writes `os.environ`; **`config.py` remains the only module that *reads*
`os.environ` for app logic** (`OPENROUTER_API_KEY` is still resolved with no
default / fail-loud inside `load_config()`, via `ConfigError` surfaced by the CLI
as a clean `error:` line + rc 2).

### How tests avoid the real `.env`

Every CLI test passes `--env-file <tmp>/does-not-exist.env`, so `load_env_file`
is a no-op and the repo `.env` is never touched. The dummy `OPENROUTER_API_KEY`
comes from the autouse conftest fixture, so `load_config()` succeeds without the
real secret. Demonstrated green with `OPENROUTER_API_KEY` unset in the shell.

## DocSpec boundary-validation-before-model-call (enforced + tested)

`main()` calls `_load_spec()` **first**, before `_resolve_mode`, before
`load_config()`, before `build_client()`, and before any `asyncio.run(...)`.
`_load_spec` reads the file, `json.loads`, then `DocSpec.model_validate`, raising
`ValueError` with a clear message on any of missing-file / bad-JSON /
schema-invalid. On failure `main` prints `error: ...` and returns 2 without ever
constructing a client or calling the controller.

Tested by `test_main_invalid_spec_rejected_at_boundary_no_model_call` (bad
`format` enum ŌåÆ rc 2, message mentions `DocSpec`, **both** `create_mode` and
`review_mode` recorders assert `.called is False`) and
`test_main_missing_required_field_rejected_at_boundary` (missing `title`). The
"no controller call" assertion is the proof that validation precedes any model
call.

## Interactive mode prompt

`_resolve_mode(mode)` returns `--mode` when given; otherwise it calls
`input("Select mode ('create', 'review'): ")` once (the only human touchpoint,
┬¦6.2) and validates the answer, raising `ValueError` on an unknown response.
`test_main_missing_mode_triggers_interactive_prompt` monkeypatches
`builtins.input` to return `create`, asserts the prompt was shown and `create_mode`
was dispatched.

## Dispatch + mockability (layering ┬¦4)

`cli.py` imports `config`, `schemas.models.DocSpec`, `core.controller`
(`create_mode`/`review_mode`), and `adapters.openrouter` to build the client ŌĆö
allowed for the composition root; nothing lower imports it. `create_mode`,
`review_mode`, and the module-level `build_client(config)` are all referenced as
module globals at call time, so tests `monkeypatch.setattr(cli, ...)` them with
async recorders / a stub, giving fully offline unit tests. `create_mode` is
`asyncio.run(...)`; `review_mode` is `asyncio.run(review_mode(document, spec,
client, config, document_name=<input stem>))`.

## Deviation / packaging note

`pyproject.toml` previously had **no** `[build-system]`, so uv treated the project
as a non-installed ("virtual") project: `pytest` imports the package via its
`pythonpath = [".", "src"]`, but `python -m thesis_agents` could not find the
package (`No module named thesis_agents`). To satisfy the acceptance criterion
"`python -m thesis_agents` runs the CLI", I added a hatchling build backend
targeting the src-layout package:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src/thesis_agents"]
```

`uv sync` then installs the package editable. Root-level `config.py` stays out of
the wheel by design (architecture ┬¦3 keeps it at the repo root); at runtime it
resolves via the cwd that `python -m` prepends to `sys.path`. `pyproject.toml` is
within feature 10's `paths`; `uv.lock` was regenerated by `uv sync`. This is the
only change beyond the two new source files and the test.
